## .github/wip/commit-pr.prompt.md

- 移動 + 変更 の差分がある場合に、これを分けようと思うと一度ロールバックする必要がある
  - 作業中に自動的にコミットしてくれる方がいい
  - 作業中のコミット粒度についての言語化がいりそう
    - 計画を事前に挟むならそのタイミングでコミット粒度まで指定できるとよさそうな気がするが、この細かさで計画すると計画外があったときの修正が果てしなくダルそう
  - コミット、PR作成を勝手にしてくれるカスタムエージェントで作業するのがいいか

## 雑記

- GitHub でリポジトリ管理するときのAI開発フロー
  - 親 Issue 作成（新機能追加、リファクタ...）
  - 親 Issue を参照させ（`gh issue view <issue_number>`）
      仕様策定
      作業計画
      タスクリスト作成
      子 Issue計画
      PR計画
      commit計画
      子 Issue 作成
    - あたりをやらせるとスプリントに組み込めそう
  - 仕様策定ファイル、作業計画ファイル（PRとの紐づけ）、タスクリストファイル（PR、コミットとの紐づけ）を親Issueに紐づくPRで作成
  - 順次 PR 作成
